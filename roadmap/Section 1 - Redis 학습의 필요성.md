
# Section 1: Redis 학습의 필요성 — 속도 차이의 이해

> Redis를 왜 배워야 하는지에 대한 근본적인 질문에 답한다. 디스크와 메모리 간의 속도 차이를 분석하고, 현대 백엔드 아키텍처에서 Redis가 필수적인 이유를 설명한다.

---

## 1. Disk vs Memory의 속도 차이

### 왜 Redis는 빠른가?

모든 것은 **데이터가 어디에 저장되느냐**에서 시작된다.

```
┌─────────────────────────────────────────────────────┐
│              데이터 접근 속도 계층 구조                │
│                                                     │
│   CPU Register    ~1ns     ██                       │
│   L1 Cache        ~1ns     ██                       │
│   L2 Cache        ~4ns     ████                     │
│   L3 Cache        ~10ns    ██████                   │
│   ─────────────────────────────────                 │
│   Main Memory     ~100ns   ██████████████  ← Redis  │
│   ─────────────────────────────────                 │
│   SSD             ~100μs   (1,000배 느림)            │
│   HDD             ~10ms    (100,000배 느림) ← RDB    │
│   Network         ~1ms+    (가변)                    │
└─────────────────────────────────────────────────────┘
```

| 저장소 | 접근 시간 | Memory 대비 |
|--------|----------|-------------|
| **Memory (RAM)** | ~100ns | 1x (기준) |
| **SSD** | ~100μs | ~1,000x 느림 |
| **HDD** | ~10ms | ~100,000x 느림 |

### 실제 성능 차이가 의미하는 것

```
상황: 사용자 프로필 조회 (1건)

PostgreSQL (Disk 기반):
  쿼리 파싱 → 실행 계획 → 디스크 I/O → 결과 반환
  약 1~10ms

Redis (Memory 기반):
  키 해시 → 메모리 직접 접근 → 결과 반환
  약 0.1~0.5ms
```

단일 쿼리에서는 차이가 미미해 보이지만, **초당 수만 건의 요청**이 들어오는 서비스에서는 이 차이가 곧 서버 증설 비용이자 사용자 이탈률이 된다.

### Disk I/O 병목 현상

전통적인 웹 서비스의 성능 병목은 대부분 **Disk I/O**에서 발생한다:

```
Client → API Server → Database (Disk I/O 병목!)
  │                      │
  │    10ms 대기          │ ← 디스크에서 데이터 읽는 시간
  │                      │
  │ ← 응답               │
```

Redis를 도입하면:

```
Client → API Server → Redis (Memory) → 0.1ms 응답
                  │
                  └→ Database (Cache Miss 시에만)
```

**핵심**: Redis는 "빠른 데이터베이스"가 아니라, **메모리에 데이터를 두어 Disk I/O를 회피하는 전략**이다.

## 2. 로컬 캐시가 있는데 왜 Redis를 쓸까?

### 로컬 캐시의 한계 — 분산 환경의 데이터 불일치

"Python의 `dict`에 캐시하면 되지 않나?" — 단일 서버에서는 맞다. 하지만 실무에서는 대부분 **여러 대의 서버**가 로드 밸런서 뒤에서 동작한다.

```
문제 상황: 서버 3대 운영

               ┌─── Server A (로컬 캐시: user:1 = "Alice")
Load Balancer ─┼─── Server B (로컬 캐시: user:1 = "Alice")
               └─── Server C (로컬 캐시: user:1 = "Alice")

사용자가 이름을 "Bob"으로 변경 (Server A에서 처리):

               ┌─── Server A (로컬 캐시: user:1 = "Bob")   ✅ 최신
Load Balancer ─┼─── Server B (로컬 캐시: user:1 = "Alice") ❌ 오래된 데이터!
               └─── Server C (로컬 캐시: user:1 = "Alice") ❌ 오래된 데이터!
```

**결과**: 같은 사용자가 요청할 때마다 다른 이름이 보인다. 이것이 **캐시 불일치(Cache Inconsistency)** 문제다.

### Redis의 해결책 — 중앙 집중 캐시

```
               ┌─── Server A ──┐
Load Balancer ─┼─── Server B ──┼──→ Redis (단일 진실의 원천)
               └─── Server C ──┘    user:1 = "Bob" ← 항상 최신

모든 서버가 같은 Redis를 바라보므로 데이터 일치 보장
```

### 로컬 캐시 vs Redis 비교

| 특성 | 로컬 캐시 (dict, LRU) | Redis |
|------|----------------------|-------|
| **속도** | 가장 빠름 (프로세스 내) | 네트워크 왕복 필요 (~0.5ms) |
| **데이터 일치** | 서버 간 불일치 | 중앙 관리로 일치 보장 |
| **용량** | 서버 메모리에 제한 | 전용 서버로 대용량 가능 |
| **프로세스 재시작** | 소멸 | 유지 (영속성 옵션) |
| **적합한 경우** | 변경 드문 정적 데이터 | 변경 잦은 동적 데이터, 분산 환경 |

> 실무에서는 **로컬 캐시 + Redis를 함께 쓰는 2-tier 캐시 전략**도 흔하다. 변경이 드문 설정값은 로컬 캐시, 변경이 잦은 세션/카운터는 Redis.

## 3. Redis의 위치: RDB와의 하이브리드 전략

### Redis는 RDB를 대체하는가?

**아니다.** Redis는 RDB를 **보완**한다.

```
┌───────────────────────────────────────────────────┐
│                   데이터 저장 전략                   │
│                                                   │
│   Redis (Memory)           RDB (Disk)             │
│   ┌─────────────┐         ┌──────────────┐        │
│   │ 자주 읽는    │         │ 모든 원본    │        │
│   │ 핫 데이터    │ ◀─ 동기 ─│ 데이터       │        │
│   │ 세션, 캐시   │         │ 트랜잭션     │        │
│   │ 카운터, 랭킹 │         │ 관계 데이터   │        │
│   └─────────────┘         └──────────────┘        │
│                                                   │
│   빠르다, 휘발 가능          느리다, 영구 보존       │
│   구조 단순                  관계/조인 가능          │
└───────────────────────────────────────────────────┘
```

### 각각의 역할

| 역할 | Redis | RDB (PostgreSQL 등) |
|------|-------|---------------------|
| **원본 데이터 저장** | ✕ (보조) | ✅ (주 저장소) |
| **트랜잭션 보장** | 제한적 | ACID 완전 지원 |
| **복잡한 조인/집계** | ✕ | ✅ |
| **캐시** | ✅ (주 용도) | ✕ |
| **세션 관리** | ✅ | 비효율적 |
| **실시간 카운터** | ✅ | 부하 큼 |
| **랭킹/리더보드** | ✅ (Sorted Set) | 쿼리 비용 높음 |
| **Pub/Sub 메시징** | ✅ | ✕ |

### 일반적인 아키텍처 패턴

```
Client
  │
  ▼
API Server
  │
  ├──→ Redis (캐시 조회, 세션, 카운터)
  │      │
  │      └── Cache Miss → DB 조회 → Redis에 저장
  │
  └──→ PostgreSQL (원본 데이터 CRUD)
```

**하이브리드 전략의 핵심**: 읽기가 많은 데이터는 Redis에서, 쓰기와 원본 관리는 RDB에서.

## 4. 글로벌 기업의 Redis 활용 사례

### Twitter/X
- **타임라인 캐싱**: 팔로우한 사람들의 최근 트윗을 Redis List로 캐싱
- **카운터**: 좋아요, 리트윗 수를 Redis INCR로 실시간 관리

### Instagram
- **피드 생성**: 팔로우 관계와 최신 게시물을 Redis에서 조합
- **세션 관리**: 수억 사용자의 로그인 세션을 Redis로 관리

### Uber
- **위치 기반 매칭**: 드라이버 위치를 Redis GEO 자료형으로 관리
- **실시간 가격 산정**: 수요/공급 데이터를 Redis에서 실시간 집계

### GitHub
- **Rate Limiting**: API 호출 제한을 Redis 카운터로 구현
- **Job Queue**: 백그라운드 작업 큐를 Redis List/Stream으로 관리

### 공통 패턴

| 패턴 | Redis 자료형 | 활용 |
|------|-------------|------|
| 캐싱 | String, Hash | DB 부하 감소 |
| 세션 | Hash + TTL | 분산 세션 관리 |
| 카운터 | String (INCR) | 조회수, 좋아요 |
| 랭킹 | Sorted Set | 리더보드 |
| 큐 | List, Stream | 비동기 작업 처리 |
| 실시간 알림 | Pub/Sub | 채팅, 알림 |
| 분산 락 | String (NX) | 동시성 제어 |
| Rate Limit | String (INCR + TTL) | API 호출 제한 |

---

## 핵심 요약

1. **Redis가 빠른 이유**: 데이터를 메모리에 저장하여 Disk I/O를 회피 (SSD 대비 ~1,000배 빠름)
2. **로컬 캐시 대신 Redis를 쓰는 이유**: 분산 환경에서 데이터 일치를 보장하는 중앙 집중 캐시
3. **Redis ≠ RDB 대체**: Redis는 RDB를 보완하는 하이브리드 전략의 핵심 구성 요소
4. **글로벌 기업의 공통 패턴**: 캐싱, 세션, 카운터, 랭킹, 큐, 실시간 알림, 분산 락, Rate Limiting
