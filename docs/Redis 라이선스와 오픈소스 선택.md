# Redis 라이선스 변경과 오픈소스 선택 가이드

## Redis 8.0 라이선스 변경

```
BSD (완전 오픈소스) → RSALv2 + SSPLv1 (제한적)
```

변경된 라이선스가 막는 건 딱 하나: Redis를 가져다가 **경쟁하는 매니지드 서비스**를 만들어 파는 것. AWS ElastiCache, GCP Memorystore 같은 사업을 견제하기 위한 변경이다.

### 영향 범위

```
영향 없음:
  ✅ 자사 서비스 백엔드에 Redis 사용
  ✅ 사내 서버에 Redis 설치해서 운영
  ✅ 개인 프로젝트
  ✅ Redis를 활용한 SaaS 제품 (Redis 자체를 파는 게 아니면)

영향 있음:
  ❌ Redis 호스팅을 서비스로 판매 (AWS가 하는 것처럼)
```

### 포크가 나온 이유

클라우드 대기업(AWS, Google)이 매니지드 서비스에 Redis를 계속 쓰려면 라이선스 비용을 내거나 포크해야 하므로 Valkey 등이 등장. 클라우드 업체들의 싸움이지 일반 개발자가 고민할 일은 아니다.

### 선택 가이드

```
개인/사내 프로젝트       → Redis 그대로 쓰면 됨
AWS 매니지드 서비스 사용  → AWS가 알아서 Valkey로 전환 중 (개발자는 동일하게 사용)
오픈소스 순수주의자       → Valkey (BSD 라이선스 유지)
```

## 반복되는 오픈소스 라이선스 변경 패턴

최근 몇 년간 같은 패턴이 반복되고 있다:

```
MongoDB      (AGPL → SSPL)     → 포크: FerretDB
Elasticsearch (Apache → SSPL)  → 포크: OpenSearch (AWS)
Redis        (BSD → RSALv2)    → 포크: Valkey
Terraform    (MPL → BSL)       → 포크: OpenTofu
```

패턴: 성공한 오픈소스 → 클라우드 업체가 무임승차 → 원작자가 라이선스 제한 → 커뮤니티가 포크.

## 라이선스 종류별 핵심 차이

| 라이선스 | 할 수 있는 것 | 주의점 |
|---------|-------------|--------|
| MIT, BSD, Apache 2.0 | 거의 다 됨 | 가장 자유로움 |
| AGPL | 사용 가능 | 네트워크로 서비스하면 소스 공개 의무 |
| SSPL | 사용 가능 | 매니지드 서비스로 파는 건 불가 |
| BSL (Business Source) | 사용 가능 | 상업적 경쟁 서비스 제한 |
| GPL | 사용 가능 | 파생 코드도 GPL로 공개 의무 |

## 개발할 때 체크 포인트

### 1. 인프라 도구 (Redis, DB 등)

서버로 별도 띄워서 네트워크로 연결하는 것은 대부분 자유롭게 사용 가능. SSPL도 문제 없다. 호스팅 서비스로 재판매하는 경우에만 SSPL/BSL이 문제된다.

### 2. 라이브러리/패키지 (npm, pip 등)

내 코드에 직접 포함(embed)되는 경우 라이선스 확인 필수. GPL이면 내 코드도 GPL로 공개 의무가 전파된다.

### 3. 이미지/폰트

상업적 사용 불가인 경우가 의외로 많다.

### 4. API 이용약관

라이선스보다 이쪽이 실무에서 더 자주 문제된다.
