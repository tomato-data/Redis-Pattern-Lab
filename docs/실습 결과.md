## Step 02: Cache-Aside 패턴

**엔드포인트**: `GET /step02/products/1`

### 첫 번째 호출 (Cache MISS)

```json
{
  "source": "database",
  "elapsed_ms": 6.008,
  "data": {
    "id": 1,
    "name": "무선 키보드",
    "price": 59000,
    "stock": 120,
    "description": "블루투스 5.0 키보드"
  },
  "cache_ttl": 328,
  "로그": "Cache MISS → DB 조회 후 캐시 저장. 6.008ms 소요"
}
```

### 두 번째 호출 (Cache HIT)

```json
{
  "source": "redis_cache",
  "elapsed_ms": 0.501,
  "data": {
    "id": 1,
    "name": "무선 키보드",
    "price": 59000,
    "stock": 120,
    "description": "블루투스 5.0 키보드"
  },
  "로그": "Cache HIT! 0.501ms 소요"
}
```

### 비교

|            | Cache MISS (DB 조회) | Cache HIT (Redis 조회) |
| ---------- | -------------------- | ---------------------- |
| source     | database             | redis_cache            |
| elapsed_ms | **6.008ms**          | **0.501ms**            |
| 속도 차이  | —                    | **약 12배 빠름**       |

SQLite(로컬 파일 DB)에서도 12배 차이가 난다. 네트워크를 타는 PostgreSQL이면 차이가 더 벌어진다.

### 동작 흐름

```
첫 번째 요청:
  Redis GET "product:1" → (nil)  Cache MISS
  → SQLite 조회 (6ms)
  → Redis SET "product:1" (TTL 300초 + jitter)
  → 응답 반환

두 번째 요청:
  Redis GET "product:1" → 데이터 있음  Cache HIT
  → 바로 반환 (0.5ms)
  → DB 조회 안 함
```

## Step 03: 최근 본 상품 (List)

**엔드포인트**: `POST /step03/{user_id}/view/{product_id}`, `GET /step03/{user_id}`

### 실행 순서

```
POST /step03/1/view/101  → [101]
POST /step03/1/view/102  → [102, 101]
POST /step03/1/view/101  → [101, 102]   ← 101이 중복 없이 맨 앞으로
```

### 목록 조회 결과 (`GET /step03/1`)

```json
{
  "command": "LRANGE recent:1 0 9",
  "user_id": 1,
  "recent_products": ["101", "102"],
  "showing": 2,
  "total": 2
}
```

### 핵심: LREM → LPUSH → LTRIM 패턴 (Pipeline)

```
1. LREM  recent:1 0 101  — 기존 중복 제거 (없으면 무시)
2. LPUSH recent:1 101    — 맨 앞에 추가 (가장 최근)
3. LTRIM recent:1 0 19   — 20개 초과분 잘라냄
```

3개 명령을 Pipeline으로 묶어 네트워크 왕복 1회로 처리한다. Pipeline 없이 하면 왕복 3회.

### 활용 사례

쿠팡, 네이버 쇼핑 등의 "최근 본 상품" 기능이 이 패턴이다.

## Step 04: 분산 세션 (Hash)

**엔드포인트**: `POST /step04/login`, `GET /step04/session-info/{session_id}`

### 로그인 후 세션 조회 결과

```json
{
  "session_key": "session:79b52a04-e269-4ba3-a2f2-62a8c409074f",
  "type": "hash",
  "data": {
    "user_id": "42",
    "email": "admin@test.com",
    "role": "admin",
    "name": "Admin"
  },
  "ttl": 1781
}
```

### 동작 흐름

```
로그인:
  UUID 생성 → session:79b52a04-...
  HSET session:{id} user_id 42 email admin@test.com role admin name Admin
  EXPIRE session:{id} 1800  (30분)
  쿠키에 session_id 저장

세션 조회 (GET /me):
  HGETALL session:{id} → 세션 데이터 반환
  EXPIRE session:{id} 1800  ← Sliding Expiration (TTL 리셋)

로그아웃:
  DEL session:{id} → 세션 즉시 삭제
```

### 핵심: Sliding Expiration

유저가 활동할 때마다 `EXPIRE`로 TTL을 리셋한다. 30분 동안 아무 요청이 없으면 세션 자동 만료, 활동 중이면 계속 유지. `type: "hash"`이므로 세션의 특정 필드(role 등)만 따로 읽거나 수정할 수 있다.

## Step 05: 조회수/좋아요 카운팅 (INCR + Set)

**엔드포인트**: `POST /step05/redis/{post_id}/view`, `POST /step05/redis/{post_id}/like`, `GET /step05/{post_id}/stats`

### 조회수 증가 (3회 반복)

`POST /step05/redis/1/view` — `INCR post:1:views`로 원자적 증가

### 좋아요 토글

```json
// 첫 번째: 좋아요
{"action": "liked", "total_likes": 1, "liked_users": ["1"]}

// 두 번째: 좋아요 취소
{"action": "unliked", "total_likes": 0, "liked_users": []}
```

Set으로 중복을 확인하므로 같은 유저가 좋아요를 두 번 누르면 취소된다.

### 전체 현황 (`GET /step05/1/stats`)

```json
{
  "redis": { "views": 3, "likes": 0 },
  "sqlite": { "views": 0, "likes": 0 }
}
```

Redis는 실시간 반영(views: 3), SQLite는 동기화 전이라 0. 이것이 **Write-Back 패턴** — 실시간 카운팅은 Redis가 담당하고, DB에는 주기적으로 동기화한다.

### 사용된 자료구조

| 용도             | 자료구조 | 키                   | 명령어                        |
| ---------------- | -------- | -------------------- | ----------------------------- |
| 조회수           | String   | `post:1:views`       | `INCR` / `DECR`               |
| 좋아요 유저 목록 | Set      | `post:1:liked_users` | `SADD` / `SREM` / `SISMEMBER` |
| 좋아요 수        | String   | `post:1:likes`       | `INCR` / `DECR`               |

## Step 06: 임시 인증번호 (TTL)

**엔드포인트**: `POST /step06/send`, `POST /step06/verify`

### 인증번호 발급

```json
{
  "phone": "010-1234-5678",
  "code_for_testing": "829386",
  "expires_in": 180,
  "commands": ["SET verification:010-1234-5678 829386 EX 180"]
}
```

### 재발송 차단 (30초 쿨다운)

```json
{ "detail": "22초 후에 재발송할 수 있습니다" }
```

### 인증 성공

```json
{
  "message": "인증 성공",
  "verified": true,
  "로그": "코드 일치 → DEL verification:010-1234-5678 (1회용이므로 즉시 삭제)"
}
```

### 핵심: TTL 하나로 세 가지 해결

| 기능           | 구현 방법                                          |
| -------------- | -------------------------------------------------- |
| 자동 만료      | `SET key code EX 180` — 3분 후 Redis가 알아서 삭제 |
| 재발송 방지    | TTL 잔여 시간으로 쿨다운 계산                      |
| 1회용          | 인증 성공 시 `DEL`로 즉시 삭제                     |
| 시도 횟수 제한 | `INCR attempts` — 5회 초과 시 차단                 |

## Step 07: 분산 락 (SET NX EX + Lua Script)

**엔드포인트**: `POST /step07/coupon/setup/{id}`, `POST /step07/coupon/{id}/issue`, `GET /step07/coupon/{id}/status`

### 쿠폰 발급 (재고 3개, 유저 3명 발급)

```json
{
  "message": "쿠폰 발급 성공! (user:3)",
  "remaining_stock": 0,
  "lock_owner": "b4e745b0-9420-44b0-b209-4453d1fbf6f6",
  "commands": [
    "SET lock:coupon:1 b4e745b0-... NX EX 30 ← 락 획득",
    "GET coupon:1:stock ← 재고 확인",
    "DECR coupon:1:stock ← 재고 차감",
    "DEL lock:coupon:1 (Lua Script로 owner 확인 후) ← 락 해제"
  ]
}
```

### 4번째 유저 시도 → 재고 소진

```json
{ "detail": "쿠폰이 모두 소진되었습니다" }
```

### 전체 현황

```json
{
  "coupon_id": 1,
  "remaining_stock": 0,
  "issued_users": ["1", "3", "2"],
  "issued_count": 3,
  "lock_active": false,
  "lock_ttl": -2
}
```

재고 3개에 정확히 3명만 발급. `lock_active: false` — 락 정상 해제 확인.

### 동작 흐름

```
1. SET lock:coupon:1 {uuid} NX EX 30  → 락 획득 (선착순 1명)
2. GET coupon:1:stock                  → 재고 확인
3. DECR coupon:1:stock                 → 재고 차감
4. Lua Script (GET + DEL 원자적)       → owner 확인 후 락 해제
```

### 핵심: Lua Script로 락 해제하는 이유

Python에서 GET → DEL을 따로 하면 두 명령 사이에 다른 요청이 끼어들 수 있다. Lua Script는 Redis 서버 내부에서 원자적으로 실행되므로 이 빈틈이 없다.

## Step 08: Rate Limiting (고정 윈도우 / 슬라이딩 윈도우)

### 고정 윈도우 (`GET /step08/fixed-window/request?user_id=1&limit=3&window=60`)

3번까지 성공, 4번째에서 차단:

```json
{
  "error": "요청 한도 초과",
  "limit": 3,
  "current": 4,
  "retry_after": 55
}
```

구현: `INCR` + `EXPIRE`. 키에 시간 단위를 포함 (`ratelimit:fixed:1:27244476`)하여 윈도우마다 자동 분리.

### 슬라이딩 윈도우 (`GET /step08/sliding-window/request?user_id=2&limit=3&window=60`)

마찬가지로 3번 성공, 4번째 차단:

```json
{
  "error": "요청 한도 초과 (슬라이딩 윈도우)",
  "limit": 3,
  "current": 3,
  "remaining": 0
}
```

구현: Sorted Set에 요청 타임스탬프를 score로 저장. `ZREMRANGEBYSCORE`로 윈도우 밖 요청 제거 후 `ZCARD`로 카운트. Lua Script로 원자적 처리.

### 두 방식 비교

|        | 고정 윈도우                              | 슬라이딩 윈도우                 |
| ------ | ---------------------------------------- | ------------------------------- |
| 구현   | `INCR` + `EXPIRE` (간단)                 | Sorted Set + Lua Script (복잡)  |
| 정확도 | 윈도우 경계에서 burst 가능 (limit의 2배) | 어느 시점이든 정확히 limit 이하 |
| 메모리 | 키 1개 (카운터)                          | 요청마다 멤버 추가              |
| 사용처 | 대략적 제한 OK인 경우                    | 정확한 제한이 필요한 경우       |

## Step 09: 실시간 랭킹 (Sorted Set)

**엔드포인트**: `POST /step09/setup`, `POST /step09/score`, `GET /step09/top`, `GET /step09/top-sqlite`

### 점수 업데이트

```json
{
  "command": "ZINCRBY leaderboard 3000 Alice",
  "player": "Alice",
  "new_score": 5500,
  "rank": 2,
  "elapsed_ms": 0.944
}
```

### Redis vs SQLite 랭킹 비교

|            | Redis (Sorted Set)   | SQLite (ORDER BY)   |
| ---------- | -------------------- | ------------------- |
| elapsed_ms | **1.287ms**          | **3.45ms**          |
| Alice 점수 | 5500 (업데이트 반영) | 2500 (동기화 안 됨) |
| 시간복잡도 | O(log N + M)         | O(N log N)          |

데이터 5건에서도 약 2.7배 차이. 데이터가 많아질수록 격차가 벌어진다.

### 핵심

- `ZINCRBY`로 점수 변경 시 자동 재정렬 (O(log N))
- Redis만 업데이트하고 DB는 동기화 안 된 상태 → Write-Back 패턴과 동일한 상황
- SQLite는 매 조회마다 `ORDER BY`로 정렬해야 하지만, Sorted Set은 항상 정렬 상태 유지

## Step 10: Pub/Sub 실시간 알림

**엔드포인트**: `POST /step10/publish`, `GET /step10/subscribe-page/{channel}`

### 구독자 있을 때

```json
{
  "command": "PUBLISH test \"hello\"",
  "receivers": 1,
  "설명": "현재 1명이 'test' 채널을 구독 중"
}
```

브라우저 구독 페이지에 실시간으로 메시지가 표시됨 (SSE).

### 구독자 없을 때

```json
{
  "receivers": 0,
  "설명": "현재 0명이 'test' 채널을 구독 중 — 구독자가 없어 메시지가 소멸됨 (Fire-and-Forget)"
}
```

### 핵심: Fire-and-Forget

구독자가 없으면 메시지는 영구 소멸. 메시지 보존이 필요하면 Stream을 사용해야 한다.

## Step 11: Stream + Consumer Group

**엔드포인트**: `POST /step11/add`, `GET /step11/range/{stream}`, `POST /step11/group/create`, `GET /step11/group/read`, `POST /step11/group/ack`, `GET /step11/group/pending`

### 메시지 추가 및 조회

```json
{ "message_id": "1774672255488-0", "stream_length": 2 }
```

`XRANGE`로 과거 메시지를 언제든 조회 가능 (Pub/Sub에서는 불가).

### Consumer Group 읽기 → ACK 흐름

```
XREADGROUP → pending_count: 2 (읽었지만 아직 처리 완료 안 함)
XACK 1개   → pending_count: 1 (ACK한 건 Pending에서 제거)
```

### Pub/Sub vs Stream 핵심 차이

```
Pub/Sub:  보냄 → 끝 (받았는지 모름)
Stream:   보냄 → 읽음(Pending) → ACK(처리 완료) → Pending에서 제거
```

워커가 메시지를 읽다가 죽어도 ACK 안 된 메시지는 Pending에 남아있어 다른 워커가 이어서 처리 가능.

## Step 12: Redis vs SQLite 성능 비교 (종합)

### 전체 벤치마크 결과

| 테스트                      | Redis   | SQLite/개별 | 차이       |
| --------------------------- | ------- | ----------- | ---------- |
| 단일 읽기                   | 0.137ms | 0.843ms     | **6.1배**  |
| 100회 반복 읽기 (avg)       | 0.088ms | 0.284ms     | **3.2배**  |
| 카운터 100회 증가           | 6.031ms | 164.063ms   | **27.2배** |
| Pipeline vs 개별 (Redis 내) | 0.724ms | 6.109ms     | **8.4배**  |
| Top 10 랭킹 조회            | 0.164ms | 1.094ms     | **6.7배**  |

### 핵심 인사이트

- **쓰기(카운터)가 가장 큰 차이** (27.2배) — SQLite는 매번 UPDATE + commit(디스크 쓰기), Redis INCR은 메모리에서 완료
- **Pipeline 효과** (8.4배) — 같은 Redis인데 네트워크 왕복만 줄여도 큰 차이
- **읽기도 차이 존재** (3~6배) — SQLite가 로컬 파일 DB임에도 메모리 기반 Redis가 일관되게 빠름
- 소규모 데이터(5건)에서 이 정도 차이이므로, 대규모 트래픽에서는 격차가 더 벌어진다

## Section 6: Redis 운영 실습

### SCAN 커서 이동 확인 (키 26개)

```
SCAN 0 MATCH user:* COUNT 5
→ 커서: 28, 결과: [user:4, user:5, user:1, user:7]

SCAN 28 MATCH user:* COUNT 5
→ 커서: 22, 결과: [user:9, user:13, user:10, user:19, user:15]

SCAN 22 MATCH user:* COUNT 55
→ 커서: 0 (순회 완료), 결과: 나머지 11개
```

### 메모리 상태

```
used_memory_human: 2.48MB (실제 사용)
used_memory_rss_human: 13.29MB (OS 할당)
maxmemory: 256MB → 전체의 1% 미만 사용
maxmemory-policy: allkeys-lru (캐시 서버 표준 설정)
```

### 영속성 설정

```
RDB: save 3600 1 / 300 100 / 60 10000 (변경량에 따라 주기 조절)
AOF: appendonly yes, appendfsync everysec (최대 1초 유실)
```

### SLOWLOG

```
slowlog-log-slower-than: 10000 (10ms)
SLOWLOG GET 5 → 비어있음 (10ms 초과 명령어 없음 = 건강한 상태)
```
